# 任务启动前确认清单（Step3 通用）

- 状态：`Approved`
- 适用范围：Step3 及后续阶段的每一次任务启动
- 使用原则：先确认、后执行；未完成清单不得开始实施

## 1. 使用方式
1. 每次新任务开始前先完整过一遍本清单。
2. 清单结果需记录到对应任务文档或阶段文档中。
3. 任一关键项不满足时，先处理阻塞项再开始任务。

## 2. 启动前必须确认项

### A. 目标与范围
- [ ] 任务目标是否明确（输入、输出、验收标准）
- [ ] 任务范围是否明确（包含项/不包含项）
- [ ] 是否已标注任务优先级与风险等级

### B. 依赖与前置
- [ ] 前置阶段是否已满足（例如 Step2 已收官）
- [ ] 依赖服务/配置是否可用（数据库、环境变量、第三方服务）
- [ ] 关联文档是否最新（索引、阶段计划、未完成清单）

### C. 风险与边界
- [ ] 是否识别可能影响线上稳定性的改动点
- [ ] 是否明确安全边界影响（认证、鉴权、审计、上传、日志）
- [ ] 是否定义不允许触碰的高风险区域

### D. 执行与回滚
- [ ] 是否定义执行步骤（最小改动路径）
- [ ] 是否定义失败回滚方案（代码、配置、数据）
- [ ] 是否明确回滚触发条件

### E. 验证与证据
- [ ] 是否明确本轮必须通过的测试清单
- [ ] 是否明确证据输出路径（`evidence/regression`、`evidence/security`）
- [ ] 是否明确文档同步范围（`00/04/05/阶段文档/证据台账`）

## 3. 任务启动记录模板

```markdown
### 任务启动确认（YYYY-MM-DD）
- 任务名称：
- 负责人：
- 目标：
- 范围：
- 风险等级：
- 前置依赖检查：通过 / 不通过（说明）
- 执行步骤：
- 回滚方案：
- 验证计划：
- 证据输出路径：
- 结论：允许启动 / 暂缓启动
```

## 4. 不满足项处理规则
1. 目标不清：先补任务定义，不得直接进入开发。
2. 依赖不可用：先解除依赖阻塞，再进入开发。
3. 回滚不明确：先补回滚方案，再进入开发。
4. 验证路径缺失：先补测试与证据计划，再进入开发。
